# Architect review — 2026-05-08

External review of PLAN.md + skeleton by the enterprise-architect agent.
Keep this file as a checklist; tick items as they're addressed and update
PLAN.md when decisions land.

---

**Verdict:** structurally clean but over-formalized for a single-binary
native app. Several real domain risks the plan doesn't acknowledge.
Phasing front-loads ceremony, back-loads the ugliest reality (mirrors,
auth, kernel sandboxing). **Recipe DSL is the biggest unforced error.**

## 1. Architectural smells

- [ ] **Service-over-Repo is Java cosplay.** `library/service.zig` is
  pure passthrough to `repository.zig`. Collapse `Service` into `Repo`
  (rename `Repo`→`Library`) until something actually orchestrates across
  repos. Same redundancy in `downloads/service.zig`. Apply YAGNI; add
  Service back when the first non-trivial use case lands.
- [ ] **Public API "re-export wall" multiplies surface, not safety.**
  `library/library.zig`, `recipe/recipe.zig`, `installer/installer.zig`
  re-list every symbol. The build wires the *module*; let callers reach
  into `domain.zig` directly. Keep one `pub const Service` and drop the
  rest. The dependency graph (build.zig) enforces layering, not the
  re-exports.
- [ ] **Vtable handlers — `canHandle` ordering is fragile.**
  `downloads/manager.zig` first-match dispatch makes registration order
  load-bearing and silent. Add `priority: u8` field, or split into a
  Resolver→handler-chain. Document or fix; don't leave landmine.
- [ ] **Anemic domain.** `library/domain.zig` has one method
  (`weightedRating`). Path-computation methods (`mergedDir()`,
  `sandboxHomeFor(game)`, `installLogPath()`) are scattered into
  Tracker/Service. Push them onto the entities.

## 2. Oversights — the dangerous ones

- [ ] **No concurrency model anywhere.** dvui drives a UI thread.
  Downloads/scrapes/convert are blocking. `manager.run()` is empty.
  Decide *now*: thread-per-job + lock-free progress channel, OR single
  worker thread + `std.Thread.Pool`. UI must read job state without
  races. dvui is immediate-mode → snapshot pattern (atomic swap of
  job-state buffer per frame). Skipping this until phase 5 forces a
  rewrite.
- [ ] **No schema migration strategy beyond ordered SQL.**
  `repository.zig` has CREATE TABLEs but no version table, no
  down-migrations, no fail-open behavior on partial migrations. PRAGMA
  `user_version` is mentioned in plan but not in code. First column
  add will lose user state.
- [ ] **No settings versioning.** `config.zig` has no `version` field.
  TOML schema *will* break.
- [ ] **No crash diagnostics path.** "No telemetry" ≠ "no story for
  panic traces." Add custom panic handler writing
  `~/.cache/f69/crashes/<ts>.log`. Zig 0.16 supports custom panic.
- [ ] **Install rollback is half-thought.** `tracker.zig` records,
  `apply.zig` is a stub and never wires Tracker around extract failures.
  Spec the semantics: every step idempotent OR wrapped in
  tempdir-then-rename. `.install.log`-as-JSONL is fine; rotation /
  concurrent-write is undefined.
- [ ] **OverlayFS in unprivileged user namespace is not portable.**
  Debian 12 default sysctl `kernel.unprivileged_userns_clone` blocks;
  Ubuntu's AppArmor blocks; nested in some containers fails silently.
  Plan says "preferred"; reality is **flat-copy will be the common case**.
  Build flat-copy *first*.
- [ ] **Sandboxie + per-game-shared-saves is best-effort, not guaranteed.**
  Boxes are stable per-name, but file namespace is COW rooted at
  `C:\Sandbox\<user>\<box>`. Sharing across versions only works if both
  versions write to same in-box paths. RPGM/Ren'Py do; obscure engines
  may not. Document.
- [ ] **No IPC abstraction for long tasks↔UI.** Same root as concurrency.
  Pick one event mechanism (single-producer-single-consumer ring per
  job, polled per frame) and put it in `util/`.
- [ ] **Resolver "learning" without explanation handling.**
  `solver.zig` mentions "learn the offending pair" — without an
  explanation tree the user gets "unsatisfied" with no cause. Cargo's
  error messages are 60% of why people accept it. Plan user-facing
  conflict explanation up front.

## 3. Better approaches

- [ ] **Drop the custom recipe DSL. Use ZON (or TOML).** The most likely
  sinkhole. ~300 LOC parser is wishful — also need lexer, validator,
  AST, error reporting, formatter (community drift). Real cost: 1500+
  LOC + forever maintenance. ZON is Zig-native, free parser via
  `std.zon.parse`, no extra dep. Anti-RCE property is a *schema* check,
  not a *grammar* property — enforce in validator regardless of format.
  **Re-spend the saved weeks on resolver UX and convert tooling.**
- [ ] **Aria2: subprocess for v1, but plan RPC from day one.** stdout-
  parsing is brittle and version-dependent.
  `--enable-rpc --rpc-listen-port=<random>` + JSON-RPC over HTTP gives
  documented progress/pause/cancel/multi-mux. Pivot now, not phase 8.
- [ ] **Collapse `installer/overlay.zig` vtable to a tagged union.** Two
  impls forever, both compiled in.
  `OverlayBackend = union(enum) { overlayfs: ..., flat: ... }`. Faster,
  simpler, exhaustive switching. Vtable is right for `downloads/handler.zig`
  (n>2, plug-in-shaped); wrong here.
- [ ] **`logging.Scope` enum is brittle.** Every new module touches a
  central enum. Use `std.log.scoped(.library_repo)` directly at call
  sites — already comptime string.
- [ ] **`f95/client.zig` rate limit needs a mutex.** Wall-clock + sleep
  is fine for v1 but two threads racing `last_request_ms` violates the
  limit silently. Wrap in `std.Thread.Mutex` *before* phase 1 ships.
- [ ] **F95 thread id as natural key.** UUIDs `[36]u8` everywhere, but
  thread id is stable, unique, integer. Saves 36 bytes/row × every join
  + simpler debugging. Drop synthetic UUID for games/mods; keep for
  installs (those need composite synthetic).

## 4. Concrete risks

- **dvui maturity.** One-maintainer, pinned to `main`. Game detail with
  mod manager + 1000-game list + drag-to-reorder + modals + settings —
  will hit gaps. **Mitigation:** prototype the busiest screen (game
  detail with mod manager) in phase 1, not phase 6. Day 7, not day 70.
- **zqlite ecosystem.** Maintained but you've traded comptime SQL
  safety for pool/blob. Wrap zqlite in a tiny project-local layer
  (`util/db.zig`) so swap-back to vrischmann is 1-day if needed.
- **bubblewrap on non-NixOS.** sysctl, AppArmor, kernel modules. Test
  matrix Debian 12 / Ubuntu 24.04 / Arch / NixOS *before* writing the
  sandbox vtable. Need `--userns-block-detected` graceful fallback.
- **Wayland/X11 inside bwrap.** Pulse, Pipewire socket, DBus session
  bus, GPU device nodes (`/dev/dri/renderD128`), `/dev/shm`, fontconfig
  cache — per-distro footguns. **Steal bwrap arg list from
  `steam-runtime-launcher-service` rather than inventing.**
- **Sandboxie automation via `Start.exe` + `Sandboxie.ini` rewrite** is
  supported but racy — `sbiectrl /reload` while another box runs
  invalidates handles. Lock around it.
- **aria2 + RPDL (private tracker).** aria2 doesn't carry F95 cookies.
  Must fetch `.torrent` via `f95/client.zig` (auth'd), *then* hand to
  aria2. Plan glosses over this.

## 5. Phasing — what to change

Current phasing optimizes for "ship a XLibrary clone by week 2" — good
for morale, **wrong for de-risking.** The novel risks (convert, sandbox,
overlay) are pushed to weeks 4-7.

Replace phasing in PLAN.md with:

1. **Week 1 — spike the three risks.** Throwaway code: bwrap a Ren'Py
   game on Debian + Arch + NixOS; flat-copy a mod over a base; convert
   one Win-only Ren'Py with `fix-linux-games.sh` logic ported. *No db,
   no UI, no DSL.* If any is harder than expected, architecture changes.
2. **Week 2 — catalog + dvui scale test.** Library + scraper, but with
   the busiest screen real (game detail + 50 mods listed). Confirms
   dvui can carry the project.
3. **Week 3 — settle recipe format (ZON), write validator, derive from
   a real F95 thread.**
4. **Week 4–5 — downloads: RPDL + auth'd `.torrent` handoff to aria2
   RPC.** Most likely reality-check; do before mods.
5. **Week 6+ — mods / resolver / overlay** — by now assumptions they
   depend on are validated.

**Bookmark import as phase 2 (3 days) is overconfident.** F95
cookie-based login + Cloudflare + bookmark page scrape is half-week
minimum, real chance of being blocked. Budget a week + "paste cookie
from browser" fallback if direct login fails.

**WS server and extension (phase 6+) correctly deferred. Keep deferred
forever if the app stays single-user.**

## What's fine — moving on

- Bounded-context layout is sound; smell is the boilerplate *inside*
  each, not the carving.
- Vtable for downloads handlers — right call.
- libsecret tokens, 127.0.0.1 WS bind — correct corrections of
  XLibrary defects.
- Per-game (not per-install) sandbox HOME — clever, correct.
- zqlite over vrischmann — correct given maintainer status.
- No telemetry / no auto-updater — correct.
- Caller-owned strings + explicit allocator — Zig-idiomatic.
